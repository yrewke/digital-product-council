# Run 2B Handoff

## What Run 2B Must Use

Package around **measured restaurant jobs**, not feature volume:

- Capture structured orders from existing demand.
- Reduce measured WhatsApp/phone leakage.
- Support profitable pickup or known fulfilment.
- Capture catering/family-tray/group enquiries.
- Provide a clear web/social/Google/QR destination.
- Measure direct repeat behavior without replacing aggregators.

## Strongest ROI-Backed Features

- Clear menu/cart and structured customer/order details.
- Stable restaurant-owned URL and source tracking.
- Pickup option and explicit fulfilment choice.
- Telegram order receiver or similarly bounded notification.
- WhatsApp handoff where customers already use WhatsApp.
- Order records sufficient to measure source/outcome.
- Catering/group enquiry capture for relevant archetypes.
- Delivery zone/fee/minimum-order controls where leakage is proven.

## Weakest Features

- Generic trust/SEO features sold as revenue generators.
- AI recommendations/assistants before demand and support economics.
- Universal loyalty/AOV uplift claims.
- Broad reporting that does not drive measured action.
- Bots without sufficient message volume.

## Core / Add-On / Delay Direction

**Core candidates:** branded page, current menu/cart, structured details, bounded notification/order record, source tracking, pickup/manual fulfilment option, clear responsibility.

**Add-on candidates:** WhatsApp handoff, deterministic WhatsApp bot after volume gate, Telegram customer bot where usage exists, catering enquiry module, payment, zone/fee rules, dashboard/status workflow, supported POS/delivery integration.

**Delay/custom:** AI assistants, complex POS/delivery, multi-branch operations, owned-fleet dispatch, advanced loyalty, open-ended automation.

## Restaurant Segments to Package For

Use these as test hypotheses, not proven fit:

1. WhatsApp-heavy restaurants with measurable manual leakage.
2. Family-tray/catering-led concepts with high-ticket enquiries.
3. Aggregator-heavy restaurants with actual statements and identifiable repeat demand.
4. Pickup-capable neighborhood QSR/grill/mandi concepts.
5. Restaurants with existing social/Google traffic but weak ordering/enquiry destination.

## Pricing Implications Without Final Public Prices

- Price and scope must follow operational responsibility and provider effort.
- Separate setup, recurring support, pass-through third-party costs, and finite additional work.
- Do not use avoided commission alone to anchor price.
- Require a provider cost scorecard: acquisition, onboarding, support, incidents, customization, and maintenance.
- Complex integrations, bots, AI, and managed operations require separate commercial gates.

## Calculator Outputs That Should Influence Sales

- Dominant measured ROI path.
- Conservative monthly net impact.
- No-break-even result.
- Missing-data and data-quality status.
- Best alternative and doing-nothing comparison.
- Best next small test.
- Operational risk and named-owner requirement.
- Fulfilment sensitivity.
- Provider-support risk.

## Exact First Question for Run 2B

Which smallest standardized package solves one measured high-value restaurant job while keeping restaurant operations, third-party costs, and one-developer support responsibility explicit and economically bounded?
