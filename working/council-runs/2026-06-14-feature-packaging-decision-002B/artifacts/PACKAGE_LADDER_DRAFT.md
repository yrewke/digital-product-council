# Package Ladder Draft

**Chairman-ready draft model:** Hybrid responsibility-based base packages plus paid modules. Prices are intentionally deferred to Run 2C.

## Tier 0 - Ordering Page

**Price:** `[Run 2C price required]`

**Job:** Capture structured orders from existing demand with the smallest real product.

**Includes:** One branded ordering page, current menu/cart, customer details/notes, Telegram order receiver, local SQLite order history, stable link.

**Restaurant owns:** Monitoring, confirmation, payment, fulfilment, complaints, refunds, menu accuracy, and staff process.

**Does not include:** Auth, dashboard, full website, payment, POS, delivery API, table ordering, advanced analytics.

**Best fit:** Disciplined small restaurant with existing repeat/WhatsApp/social demand and a named order owner.

## Tier 1 - Website + Ordering

**Price:** `[Run 2C price required]`

**Job:** Provide a clear owned destination for Google/social/QR attention and capture structured orders.

**Includes:** Tier 0 plus template-based home page, contact, hours, location/map link, social links, QR code, basic structured information, source-tag capability.

**Does not include:** Staff dashboard, payment, POS/delivery integration, managed marketing, guaranteed SEO/results.

**Best fit:** No/weak-website restaurant with visible demand, Instagram/Google activity, family/group enquiry potential.

## Tier 2 - Direct Order Workflow

**Price:** `[Run 2C price required]`

**Job:** Reduce measured manual-order leakage and create one accountable direct-order workflow.

**Includes:** Tier 1 capabilities as needed, staff auth/dashboard, order queue, accept/reject/status flow, order history, availability/menu management, pickup and manual delivery details.

**Restaurant owns:** Staff discipline, food/fulfilment, manual POS entry, complaints/refunds unless separately scoped.

**Does not include:** Guaranteed POS integration, delivery API, online payment, 24/7 managed operations.

**Best fit:** WhatsApp-heavy, higher-volume, pickup/manual-delivery restaurant with a named operator.

## Commerce Modules After Tier 2

**Price:** `[Run 2C module prices required]`

**Job:** Add one bounded commerce capability where its economics and responsibility are known.

**Available modules:** Payment, delivery zones/fees/minimums, customer records/saved addresses, controlled promotions, and deeper source/outcome analytics.

**Boundary:** There is no fixed all-in Tier 3 bundle in the current packaging decision. Each commerce module requires restaurant data, provider compatibility, explicit responsibility, and separate pricing. Supported integrations remain separate modules/custom quotes.

**Candidate fit:** Proven Tier 2 workflow with known fulfilment/payment economics.

## Future Operations / Integration

**Price:** `[custom quote]`

Kitchen view, multi-branch, advanced roles/reporting, supported integrations beyond standard adapters, webhooks, complex migration, and other operational work. No universal promise.

## Launch Validation Motion

Every early sale may use a standardized, time-bounded launch-validation plan attached to the selected real package. It measures demand source, usage, operational burden, and provider effort. It is not a fake Tier 0 or indefinite consulting engagement.
